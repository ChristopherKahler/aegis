<objective>
## Goal
Define the complete AEGIS framework architecture and file type conventions. Produce standards documentation that governs how every subsequent artifact is structured, named, cross-referenced, and versioned.

## Purpose
AEGIS is a prompt-based epistemic engine with 8 distinct component types. Without formal conventions for each type, subsequent phases will produce inconsistent, unauditable, non-composable artifacts. This plan establishes the architectural foundation — the "constitution" that all future work must conform to.

## Output
- `docs/ARCHITECTURE.md` — Master architecture document (directory structure, component model, cross-referencing, naming, version-locking)
- `docs/standards/*.md` — One convention specification per component type (8 files)
- `docs/standards/runtime.md` — Per-audit runtime structure and versioning specification
</objective>

    Create the AEGIS architecture document covering:

    **1. Directory Structure**
    Define the complete framework layout:
    ```
    aegis/
    ├── src/
    │   ├── commands/        # User entry points (/aegis:*)
    │   ├── personas/        # 12 core persona definitions
    │   ├── domains/         # 14 domain knowledge modules
    │   ├── schemas/         # Output format contracts
    │   ├── rules/           # Epistemic governance
    │   ├── tools/           # Tool adapter configs + normalizers
    │   ├── agents/          # Assembly manifests (composition definitions)
    │   └── workflows/       # Phase orchestration logic
    ├── docs/
    │   ├── ARCHITECTURE.md  # This file
    │   ├── standards/       # Convention specs per component type
    │   ├── GETTING-STARTED.md
    │   └── USER-GUIDE.md
    ├── README.md
    └── .aegis-template/     # Copied into target repo per audit
    ```
    Explain each directory's purpose and what file types it contains.

    **2. Component Model**
    Document the 8 component types in a table: type, count, purpose, what it defines, what references it.
    Then explain the decomposed agent architecture:
    - Agent = Persona + Domains[] + Schemas + Rules + Tool Interfaces
    - Three-layer design: Core Persona (strong/distinct) → Domain Modules (neutral/structured) → Execution Envelope (shared contracts)
    - Why monolithic agents are wrong for this system

    **3. Component Relationships**
    Document the relationship matrix:
    - Agents ↔ Domains (many-to-many)
    - Domains ↔ Tools (many-to-many)
    - Agents → Schemas (agents produce findings using schemas)
    - Workflows → Agents (orchestration invokes agents)
    - Rules → Agents (rules constrain all agent behavior)
    - Commands → Workflows (entry points delegate to orchestration)

    **4. Cross-Referencing Patterns**
    Define how components reference each other:
    - @-reference syntax for file loading
    - Assembly manifest syntax (how agent files declare their components)
    - Domain mapping syntax (how agents declare which domains they cover)
    - Tool affinity syntax (how domains declare which tools feed them)

    **5. Naming Conventions**
    Define naming rules for:
    - Files: kebab-case (e.g., security-engineer.md)
    - Directories: kebab-case
    - Component IDs: snake_case identifiers in frontmatter/headers
    - Domain numbering: DD format (00-13)
    - Agent numbering: follow README order (01-12)

    **6. Version-Locking**
    Define how audit runs record component versions:
    - Manifest format listing component type + file + hash/version
    - Where the manifest lives in the audit output
    - How to reproduce an audit with locked versions

    Reference PAUL framework patterns for structural inspiration but derive all conventions from AEGIS's specific needs. Do NOT copy PAUL structure blindly — justify every convention from AEGIS's requirements.
  </action>

    Create 8 convention specification files, one per component type. Each file follows a consistent meta-structure:

    ```
    # [Type Name] Convention

    ## Purpose
    What this component type is and why it exists in AEGIS.

    ## Location
    Where files of this type live in the directory structure.

    ## Naming
    File naming pattern with examples.

    ## Required Structure
    Exact tags, frontmatter fields, and sections a conforming file must have.
    Include which elements are required vs optional.

    ## Cross-References
    How this type references and is referenced by other types.

    ## Example Skeleton
    Complete skeleton of a conforming file using placeholder conventions:
    - [Square brackets] for human-fillable content
    - {Curly braces} for variable interpolation

    ## Anti-Patterns
    Common mistakes with rationale for why they're wrong.
    ```

    **Per-type guidance:**

    **personas.md** — Convention for persona files (12 total).
    Study the README's agent descriptions to understand what persona files must capture: identity, role, mental models, risk philosophy, thinking style, what triggers concern, how they argue, what they ignore, confidence calibration, "must never" constraints. Personas must be STRONG and DISTINCT — the convention must prevent persona dilution. Consider: frontmatter (id, name, role, active_in_phases), semantic XML tags for identity sections. Anti-pattern: embedding domain-specific knowledge (that belongs in domain files).

    **domains.md** — Convention for domain knowledge modules (14 total, numbered 00-13).
    Study README domains 0-13. Each domain module captures: failure patterns, audit questions, red flag indicators, tool affinities, relevant standards/frameworks, metrics. Domains must be NEUTRAL and STRUCTURED — the convention must prevent opinion from leaking in. Consider: frontmatter (id, number, name, owner_agents[]), consistent internal sections. Anti-pattern: embedding agent-specific reasoning style (that belongs in persona files).

    **schemas.md** — Convention for output format contracts (~5: finding, disagreement, confidence, signal, report-section).
    Study README's epistemic schema, disagreement model, and confidence modeling sections. Schemas are documents that agents fill out — they define structure, field types, enums, and validation rules. Consider: template-style format (fenced code block with the schema, then field documentation tables). Anti-pattern: embedding interpretation guidance (that belongs in rules or personas).

    **rules.md** — Convention for epistemic governance rules.
    Study README's 5 epistemic hygiene rules and the disagreement anti-patterns. Rules are global constraints that apply to ALL agents. Consider: frontmatter with scope (which component types the rule applies to), prescriptive DO/DON'T format. Anti-pattern: rules that are actually domain knowledge or persona traits.

    **tools.md** — Convention for tool adapter specifications (7+ tools).
    Study README's tool stack and signal dimensions. Each tool spec covers: what it does, how to run it, what output format it produces, how to normalize output to AEGIS signal format, which domains it feeds. Consider: frontmatter (id, name, type, domains_fed[], install_required), sections for config, execution, normalization. Anti-pattern: embedding audit logic (tools produce signals, agents interpret them).

    **agents.md** — Convention for assembly manifest files (12 total).
    These are THIN composition files — they declare which persona, which domains, which tool feeds, which schemas to use. They are NOT prompts. They are assembly instructions that workflows use to compose the runtime agent. Consider: primarily frontmatter/YAML-heavy format (persona_ref, domains[], tools[], schemas[], rules[], active_phases[]). Anti-pattern: duplicating persona content or domain knowledge.

    **workflows.md** — Convention for orchestration workflow files (~6: one per AEGIS execution phase + utility flows).
    Study README's 6 execution phases and PAUL's workflow patterns. Workflows define step-by-step execution logic for multi-session orchestration. Consider: similar to PAUL workflows but adapted for multi-session agent invocation (<purpose>, <phase_context>, <step> tags, session handoff specifications). Anti-pattern: embedding domain knowledge or persona behavior.

    **commands.md** — Convention for user-facing slash command files (~4: /aegis:audit, /aegis:resume, /aegis:status, /aegis:report).
    Study PAUL's command patterns. Commands are thin wrappers that provide guided wizard UX. Consider: YAML frontmatter (name, description, argument-hint), <objective>, <execution_context> referencing workflows, guided option presentation format. Anti-pattern: embedding execution logic (that belongs in workflows).

    Each convention file must be self-contained — readable and actionable without needing to read the other 7. But cross-references between types should use the patterns defined in ARCHITECTURE.md.
  </action>
