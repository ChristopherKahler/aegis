## Design Source
The Transform architecture was designed in a product discussion (this session). Key design elements:

### Two-System Architecture
```
AEGIS Core (Diagnosis Engine)
  Phase 0-5: Existing audit pipeline
  Output: Layer A — Immutable diagnostic artifacts

AEGIS Transform (Controlled Evolution Engine)
  Phase 6: Remediation Synthesis
  Phase 7: Change Risk Validation
  Phase 8: Execution Planning → PAUL handoff
  Output: Layer B — Remediation knowledge (human + machine)
  Output: Layer C — Change orchestration (PAUL project)
```

### Three Output Layers
- **Layer A: Diagnostic Artifact (Truth Layer)** — Immutable, reproducible, epistemically versioned. The audit itself. Never mutated.
- **Layer B: Remediation Knowledge Layer (Instruction Layer)** — Derived from findings. Framework-specific. Pattern-based. Educational. Where playbooks live. Dual format: human-readable markdown + machine-consumable structured data.
- **Layer C: Change Orchestration Layer (Execution Layer)** — Dependency-aware plan. Change graph. Risk scoring. Verification gates. This is operational sequencing, not documentation. PAUL framework is the integration point for this layer.

### Four Intervention Levels
Every Transform output must be explicitly classified:
1. **Suggesting** — "Consider this pattern" (informational, no action implied)
2. **Planning** — "Here's how to fix this" (structured plan, human decides)
3. **Authorizing** — "This change is recommended with confidence X" (risk-scored, gate-checked)
4. **Executing** — "Apply this change" (only via PAUL with verification gates, never auto-applied by AEGIS)

### 4-Layer Transformation Model
Remediation knowledge is parametric, not generic:
```
Layer 1: Abstract Pattern     → "Unbounded retries are dangerous"
Layer 2: Framework Mapping    → "In Laravel, use retryUntil()"
Layer 3: Language Mapping     → "PHP retry patterns with exponential backoff"
Layer 4: Project Context      → "Files X, Y, Z — these 3 call sites"
```

### 5 Transform Agent Personas
Diagnosis is decentralized (8+ domain agents work independently). Intervention is centralized (5 Transform agents coordinate):

| Agent | Role | Consumes |
|-------|------|----------|
| Remediation Architect | Translates diagnosis → structured change plan | All findings + domain knowledge |
| Change Risk Modeler | Scores blast radius, coupling, regression probability, architectural tension | Change plan + codebase structure |
| Pedagogy Agent | Explains fixes for AI-assisted developers (vibe coders) | Remediation plan + framework context |
| Guardrail Generator | Writes project rules (.claude/CLAUDE.md, .cursorrules) for future AI usage | Patterns + findings |
| Execution Validator | Defines verification plans — how to prove fixes work | Change plan + test infrastructure |

### Safety & Liability Framework
Moving from Advisor to Architectural Actor requires:
- Conservative bias (default to suggesting, not executing)
- Confidence thresholds (do not generate remediation if finding confidence < threshold)
- Unsafe context flagging (flag when change risk exceeds acceptable bounds)
- Intervention level gating (higher levels require higher confidence + lower change risk)

### Change Risk in Remediation
Remediation introduces new risk. The system must model:
- Blast radius (how much breaks if fix is wrong)
- Coupling risk (does fix create new dependencies)
- Regression probability (does fix break existing behavior)
- Architectural tension (does fix fight the existing design)

### Feedback Loop
Anti-pattern → Correct pattern → Verified improvement becomes a proprietary failure-pattern corpus over time. The system should be designed to accumulate verified architectural migrations.

### PAUL Integration
Layer C (Change Orchestration) generates PAUL-ready project artifacts:
- Phased remediation plans with dependency ordering
- Verification gates per change
- Risk-scored task definitions
- AEGIS Transform does NOT execute changes — it hands off to PAUL, which the user's AI assistant executes
