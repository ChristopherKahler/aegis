<objective>
## Goal
Create the three core output schemas that define how AEGIS agents produce structured diagnostic findings: the 7-layer epistemic finding schema, the disagreement record schema, and the confidence vector schema.

## Purpose
These schemas are the fundamental data contracts of AEGIS Core (Layer A). Every agent finding, every disagreement, and every confidence assessment must conform to these structures. Without them, agent output drifts — severity scales diverge, evidence quality varies, and disagreement resolution has no structured input. These three schemas make the entire system composable and epistemically rigorous.

## Output
Three schema files in `src/schemas/`, each conforming to the schema convention (`docs/standards/schemas.md`):
- `finding.md` — 7-layer epistemic finding with full template, field reference, validation rules, and worked example
- `disagreement.md` — Structured disagreement record with positions, root cause taxonomy, resolution models
- `confidence.md` — 4-dimension confidence vector with scoring system and calibration guidance
</objective>

  <files>src/schemas/finding.md</files>
  <action>
    Create `src/schemas/` directory and the finding schema file.

    **Frontmatter:**
    - id: finding
    - name: Epistemic Finding
    - version: 1.0.0
    - used_by: list all 12 Core agent IDs + principal-engineer (all produce findings)

    **Purpose section:**
    - A Finding is the atomic unit of agent output — a single identified issue, risk, or observation
    - Every concern an agent raises must be expressed as a Finding
    - Findings feed into disagreement resolution, severity calibration, and report generation
    - The 7-layer structure enforces epistemic decomposition — facts separated from interpretation separated from judgment

    **Template section:**
    Build the template encoding all 7 epistemic layers from README lines 912-982:
    - Layer 1 (Observation): Raw signal, no adjectives, no risk language
    - Layer 2 (Evidence Source): source_type, tool_or_artifact, location, freshness
    - Layer 3 (Interpretation): Causal mechanism, multiple interpretations allowed
    - Layer 4 (Assumptions): What must be true for interpretation to hold
    - Layer 5 (Risk Statement): "If [interpretation], then [failure mode], impacting [asset]"
    - Layer 6 (Impact & Likelihood): impact_domain, impact_magnitude, likelihood, time_horizon, blast_radius
    - Layer 7 (Judgment): action recommendation, owning_agent (Principal owns final judgment)

    Also include metadata fields:
    - finding_id: Format F-{DD}-{NNN} (DD = domain number, NNN = sequence)
    - domain: number + name
    - agent_id: producing agent
    - severity: critical | high | medium | low | informational
    - confidence_vector: reference to confidence.md schema (4-dimension vector)

    **Field Reference:**
    Table documenting every field with: Field, Type, Required, Description, Valid Values
    - Include all 7-layer fields plus metadata fields
    - Include all enum values explicitly (severity, likelihood, impact_magnitude, etc.)

    **Validation Rules:**
    Encode epistemic integrity constraints:
    1. finding_id uniqueness across entire audit
    2. domain_number must be 00-13
    3. agent_id must match an agent with the specified domain
    4. Layer 1 must contain no adjectives or risk language (observation only)
    5. Layer 2 must contain at least one concrete source reference
    6. Layer 3 must explain a causal mechanism (not just restate observation)
    7. Layer 4 must list at least one falsifiable assumption
    8. Layer 5 must follow "If [X], then [Y], impacting [Z]" structure
    9. Layer 6 fields must all use their enumerated values
    10. Layer 7 judgment must be one of: must_fix, should_fix, accept_risk, monitor, out_of_scope
    11. confidence_vector must be a valid confidence schema instance
    12. severity and confidence_vector must be consistent (critical severity + low confidence triggers review)

    **Examples section:**
    One complete worked example: a realistic security finding (hardcoded credentials) showing all 7 layers filled correctly. Use the finding skeleton from docs/standards/schemas.md as a starting point but expand it with full 7-layer detail.

    **Avoid:**
    - Embedding workflow logic or behavioral rules (those belong in rules files)
    - Including Transform-specific fields (playbook references belong in playbook schema)
    - Making fields optional that should be required — drift starts with optional fields
  </action>

    - agents_involved: List of agent IDs participating in the disagreement
    - positions: Array of position objects, each containing:
      - agent_id: Which agent holds this position
      - claim: What the agent asserts
      - evidence: Evidence supporting the claim
      - assumptions: Assumptions underlying the position

    - resolution_model_applied: One of the 5 models (README lines 1022-1042):
      evidence_dominance, risk_asymmetry, reversibility, time_to_failure, blast_radius
    - principal_response: REQUIRED — Principal Engineer's explicit response (silence is not allowed)
    - status: One of: open, mitigated, accepted_risk, deferred, out_of_scope

    **Field Reference:**
    Table documenting every field including nested position fields.

    **Validation Rules:**

    2. finding_id must reference an existing finding

<task type="auto">
  <name>Task 3: Create the confidence vector schema</name>
  <files>src/schemas/confidence.md</files>
  <action>
    Create the confidence vector schema file.

    **Frontmatter:**
    - id: confidence
    - name: Confidence Vector
    - version: 1.0.0
    - used_by: [all agents — every finding carries a confidence vector]

    **Purpose section:**
    - Confidence in AEGIS is a vector, not a scalar
    - A single "high/medium/low" hides which aspects of confidence are strong vs weak
    - The 4-dimension model enables nuanced statements like "High-impact, low-confidence risk — validate before remediation"
    - Dimensions are independently scored, producing a profile rather than a rating
    - Transform intervention levels gate on confidence dimensions (not just aggregate confidence)

    **Template section:**
    Build from README lines 984-994 with detailed scoring:
    - evidence_diversity: How many independent sources support this finding
      - 1: Single tool output, no corroboration
      - 2: Single source type (e.g., two static analysis tools)
      - 3: Two source types (e.g., static + config analysis)
      - 4: Three+ source types (e.g., static + runtime + history)
      - 5: Multiple independent source types with cross-validation
    - signal_freshness: Temporal relevance of the evidence
      - 1: Historical only (no current evidence)
      - 2: Static analysis only (code as-written, not as-running)
      - 3: Static + recent configuration/deployment data
      - 4: Includes recent runtime or log data
      - 5: Live/current runtime validation available
    - assumption_fragility: How easily the finding's assumptions could be wrong
      - 1: Finding depends on multiple unverified assumptions
      - 2: Key assumption is plausible but unverified
      - 3: Key assumptions are reasonable with partial evidence
      - 4: Most assumptions are verified or low-risk
      - 5: All assumptions are verified or self-evident
    - historical_precedent: Whether this pattern is a known failure mode
      - 1: Novel pattern, no known precedent
      - 2: Theoretically possible failure mode
      - 3: Known anti-pattern in literature
      - 4: Documented failure mode with real-world incidents
      - 5: Frequent failure pattern with extensive incident history
    - overall: Derived aggregate (NOT a simple average — formula defined in validation rules)
    - justification: Required text explaining the vector scores

    **Field Reference:**
    Table documenting each dimension, the overall field, and the justification field.

    **Validation Rules:**
    1. Each dimension must be an integer 1-5
    2. overall is calculated as: min(evidence_diversity, signal_freshness) weighted by assumption_fragility
      - Formula: overall = floor((min(evidence_diversity, signal_freshness) * 0.6) + (assumption_fragility * 0.3) + (historical_precedent * 0.1))
      - Map to enum: 1-2 = low, 3 = medium, 4-5 = high
      - Note: This formula is conservative by design — weakest link in evidence drags overall down
    3. justification is required (not optional) — at minimum one sentence per dimension scored 4 or 5
    4. A confidence vector where any dimension is 1 caps overall at "low" regardless of other dimensions
    5. evidence_diversity of 1 with any dimension at 5 is flagged as inconsistent (single source can't produce highest scores elsewhere)
    6. Confidence vectors are immutable once attached to a submitted finding (revisions require a new finding version)

    **Examples section:**
    Two worked examples:
    1. High-confidence finding: Hardcoded credentials detected by multiple tools, well-known anti-pattern
    2. Low-confidence finding: Potential race condition inferred from code structure alone, no runtime validation

    **Avoid:**
    - Reducing to a single scalar (the whole point is the vector)
    - Making justification optional (unjustified high confidence is the #1 epistemic failure mode)
    - Over-complicated formulas — the overall derivation should be simple enough to compute mentally
