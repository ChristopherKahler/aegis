    **Signal Schema (src/schemas/signal.md):**

    Frontmatter: id: signal, name: Signal, version: 1.0.0, used_by: [tool adapters, all audit agents]

    Purpose: The signal schema normalizes heterogeneous tool output into a uniform format. Tools speak different languages — SonarQube reports "blocker/critical/major/minor", Semgrep uses "ERROR/WARNING/INFO", Trivy uses "CRITICAL/HIGH/MEDIUM/LOW". The signal schema converts all tool output into four normalized dimensions so agents can compare and correlate across tools.

    Template fields:
    - signal_id: Format S-{TTT}-{NNN} (TTT = tool abbreviation, NNN = sequence). Example: S-SMG-001
    - source_tool: Tool that produced this signal (e.g., "semgrep", "sonarqube", "trivy")
    - source_rule: The specific rule/check that fired (e.g., "python.security.sql-injection")
    - raw_output_ref: Path to the raw tool output file for traceability
    - location: File path + line range where the signal was detected
    - signal_category: One of the 6 signal dimensions from README: structure, behavior, history, dependencies, policy_posture, runtime_contracts
    - severity: Normalized to AEGIS scale: critical, high, medium, low, informational
    - confidence_estimate: Simplified scalar (high/medium/low) — NOT the full confidence vector (that's computed by agents after analysis)
    - blast_radius: Normalized estimate: localized, service_level, systemic, org_wide
    - domain_relevance: List of domain numbers (00-13) this signal is relevant to
    - raw_severity: The original severity string from the tool (preserved for audit trail)
    - normalization_notes: Any context about how raw values were mapped to AEGIS values

    Validation rules:
    1. signal_id must be unique across the audit
    2. source_tool must match a configured tool adapter
    3. severity must use AEGIS enum values
    4. domain_relevance must contain at least one valid domain number
    5. raw_severity must be preserved (never discarded)
    6. confidence_estimate is a tool-level estimate, not agent-level confidence (agents compute full confidence vectors)

    Example: Normalize a Semgrep SQL injection finding into the signal schema.

    **Report Section Schema (src/schemas/report-section.md):**

    Frontmatter: id: report-section, name: Report Section, version: 1.0.0, used_by: [principal-engineer, workflows]

    Purpose: Defines the structure of the Layer A diagnostic report — the primary AEGIS deliverable. The report has 7 sections in fixed order. Each section has a defined purpose, required content, and data sources.

    Template: Define a report as an ordered collection of sections. Each section type specifies:
    - section_number: 1-7
    - section_name: From the README's 7 report sections
    - content_requirements: What must be in this section
    - data_sources: Which schemas/artifacts feed this section
    - audience: Who reads this section

    The 7 sections:
    1. Executive Risk Summary — One-page leadership overview. Sources: aggregated findings by severity, top disagreements.
    2. Architecture Narrative — How the system is built. Sources: Domain 01 findings, context from Phase 0.
    3. Findings by Domain (Severity-Ranked) — All 14 domains. Sources: @schema:finding instances grouped by domain, sorted by severity.
    4. Cross-Validation Notes — All disagreements and resolutions. Sources: @schema:disagreement instances.
    5. Remediation Roadmap — Prioritized action plan. Sources: findings with action: must_fix or should_fix, ordered by severity and dependency.
    6. Long-Term Structural Risks — Degradation over time. Sources: findings with time_horizon: long_term or hypothetical.
    7. "What Would Break First at 10x Scale" — Predictive failure analysis. Sources: Domain 08 (Performance), Domain 07 (Reliability) findings, blast_radius: systemic findings.

    Validation rules: All 7 sections required, sections must be in order, each section must reference its data sources.

    Example: One complete report section (Executive Risk Summary) with realistic content.

    **Avoid:**
    - Don't define Transform report sections (Layer B/C) — those are Plan 02-03 scope
    - Don't embed workflow logic about how reports are generated — that's Phase 6 scope
  </action>

    **Epistemic Hygiene Rules (src/rules/epistemic-hygiene.md):**

    Frontmatter: id: epistemic-hygiene, name: Epistemic Hygiene, scope: all_agents, priority: critical

    Purpose: Without these rules, agents produce findings that sound confident but lack evidence, assert severity without justification, and treat tool output as ground truth. These 5 invariants are non-negotiable — they are the bedrock of AEGIS's epistemic integrity.

    Rules (from README lines 996-1005):
    1. "No risk statements without observations" — Layer 5 (risk) requires Layer 1 (observation). Enforcement: Finding validation rejects risk statements where Layer 1 is empty or vague.
    2. "No judgments without risk modeling" — Layer 7 (judgment) requires Layer 6 (impact/likelihood). Enforcement: Finding validation rejects judgments where Layer 6 fields are missing or use non-enum values.
    3. "No confidence without evidence" — Confidence vector requires Layer 2 (evidence source). Enforcement: Confidence validation rejects vectors where evidence_diversity > 1 but Layer 2 lists only one source.
    4. "No synthesis without acknowledging uncertainty" — Report synthesis sections must reference assumption fragility and open disagreements. Enforcement: Report validation checks that Executive Summary and Remediation Roadmap reference uncertainty.
    5. "No 'clean narrative' without Devil's Advocate response" — Phase 5 report cannot be finalized without Devil's Advocate critique being addressed. Enforcement: Report generation workflow checks for unresolved Devil's Advocate disagreements.

    DO section: 3-4 concrete examples of conforming behavior.
    DON'T section: 3-4 concrete violations with explanations of why each is wrong.

    **Disagreement Protocol Rules (src/rules/disagreement-protocol.md):**

    Frontmatter: id: disagreement-protocol, name: Disagreement Protocol, scope: all_agents, priority: critical

    Purpose: Multi-agent systems are prone to false consensus. Without explicit protocol rules, disagreements get auto-resolved, averaged, or hidden. These rules ensure that every disagreement is a first-class object that must be explicitly addressed.

    Rules (from README lines 1084-1092):
    1. "No auto-resolving disagreements" — Disagreements cannot be marked as resolved without Principal response. Enforcement: Disagreement schema validation rejects status != "open" when principal_response is empty.
    2. "No averaging opinions" — Resolution cannot split the difference (e.g., "severity is between high and medium"). Enforcement: Principal response is checked for averaging language patterns.
    3. "No forcing consensus language" — Resolution cannot claim agents now agree when positions haven't changed. Enforcement: Resolution that claims consensus must show updated positions.
    4. "No hiding disagreement in footnotes" — Every disagreement referenced in findings must have a corresponding disagreement record. Enforcement: Cross-reference validation between findings and disagreement records.
    5. "No treating Devil's Advocate as optional" — Devil's Advocate critique must be produced for every audit. Dismissing it as out_of_scope without substantive rationale is a violation. Enforcement: Report generation checks for Devil's Advocate disagreement records.

    DO section: 3-4 concrete examples.
    DON'T section: 3-4 concrete violations with explanations.

    **Avoid:**
    - Don't include domain-specific knowledge (that's domain files, not rules)
    - Don't include persona-specific traits (that's persona files)
    - Don't duplicate schema validation (schemas enforce data format; rules enforce behavior)
  </action>

    **Agent Boundary Rules (src/rules/agent-boundaries.md):**

    Frontmatter: id: agent-boundaries, name: Agent Boundaries, scope: all_agents, priority: critical

    Purpose: AEGIS agents are composed from personas + domains + schemas + rules. The power of the decomposed architecture depends on agents staying within their boundaries. When an Architect agent starts opining on security, or a Security agent starts producing performance recommendations, the analysis loses its structural integrity. Boundary rules prevent agents from overstepping their compositional contracts.

    Rules:
    1. "Persona identity constraint" — An agent must operate within its persona's defined thinking style, risk philosophy, and mental models. An SRE agent must not adopt a Security Engineer's threat model. Enforcement: Review of findings checks that reasoning patterns match declared persona characteristics.

    2. "Domain scope constraint" — An agent must produce findings only in domains listed in its agent assembly manifest's `domains` field. A Security Engineer (Domain 4) must not produce findings in Domain 08 (Performance). Enforcement: Finding validation checks that finding domain_number is in the agent's declared domain list.

    3. "Cross-domain observation, not cross-domain judgment" — An agent may observe signals outside its domains (e.g., a Security Engineer may note that a performance issue creates a DoS vector), but the finding must be filed under the agent's own domain with a cross-reference to the relevant domain. Enforcement: Cross-domain observations must use the agent's domain number and include a reference note.

    4. "Schema conformance constraint" — All agent output must conform to the schemas declared in the agent's assembly manifest. No ad-hoc output formats. Enforcement: Output validation checks every finding against @schema:finding, every disagreement against @schema:disagreement.

    5. "Tool signal consumption constraint" — Agents consume signals tagged with their domains (domain_relevance field in @schema:signal). Agents should not ignore signals relevant to their domains, and should not give undue weight to signals irrelevant to their domains. Enforcement: Audit trail checks signal consumption against agent domain assignments.

    DO section: 3-4 concrete examples of proper boundary behavior.
    DON'T section: 3-4 concrete violations with explanations.

    **Avoid:**
    - Don't define what each persona's identity IS (that's persona files)
    - Don't list which domains each agent has (that's agent assembly manifests)
    - Rules define the CONSTRAINTS — other component types define the specifics
  </action>
