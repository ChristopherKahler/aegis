<objective>
## Goal
Create the 4 Transform schemas and 2 Transform safety rule files that govern AEGIS Transform's remediation planning, change-risk assessment, intervention governance, and safety constraints.

## Purpose
Transform artifacts control how AEGIS evolves codebases — moving from diagnosis (Layer A) to remediation knowledge (Layer B) and change orchestration (Layer C). Without schemas, Transform output is unstructured and unvalidatable. Without safety rules, Transform can produce overconfident or unsafe remediation plans. These artifacts complete the data contracts and behavioral constraints for the entire AEGIS system.

## Output
- 4 Transform schema files in `src/transform/schemas/`
- 2 Transform safety rule files in `src/transform/rules/`
- Total: 6 files completing Phase 2 (Schemas & Rules)
</objective>

  <name>Task 3: Create Transform safety rules</name>
  <files>src/transform/rules/safety-governance.md, src/transform/rules/change-risk-rules.md</files>
  <action>
    Create directory `src/transform/rules/`.

    Organize the 6 safety categories from docs/standards/rules.md into 2 rule files:

    **safety-governance.md** — Intervention governance and execution boundaries:
    - Frontmatter: id: safety-governance, name: Safety Governance, scope: transform_agents, priority: critical
    - Purpose: Explain what goes wrong without governance — overconfident remediation, unsafe auto-execution, liability without evidence, intervention escalation without justification. These rules are the hard boundaries of Transform.
    - Rules (covering 4 categories from standards):
      1. No Auto-Execution — Transform NEVER applies changes to codebases. Produces plans only. No bypass, no trusted mode, no override. (From category 4)
      2. Conservative Bias — Default to lowest intervention level that serves the user. When uncertain, downgrade. Never escalate without evidence. (From category 1)
      3. Confidence Gating — Minimum finding confidence per intervention level: Suggesting=any, Planning=medium+2 sources, Authorizing=high+3 cross-validated, Executing=high+3 cross-validated. (From category 2)
      4. Liability Escalation — Suggesting/Planning = Advisor posture. Authorizing/Executing = Architectural Actor posture. Higher liability requires higher confidence and lower risk. Authorizing-level must include disclaimer. (From category 6)
      5. Intervention Level Immutability — Once a playbook is assigned an intervention level and approved at that level, the level cannot be upgraded without re-assessment. Downgrading is always permitted. (New — prevents post-approval escalation)
    - Each rule: statement, rationale (what failure mode it prevents), enforcement (concrete mechanism referencing @schema:intervention-level and @schema:playbook)
    - DO section: 4+ concrete examples of conforming behavior (e.g., "Playbook assigned Planning level with note: 'Evidence from 2 sources (Semgrep + manual review). Confidence medium. Cannot justify Authorizing without third independent source.'")
    - DON'T section: 4+ violations with explanations (e.g., "Playbook assigned Authorizing level for a finding with confidence: low" — why wrong: "Authorizing requires high confidence with 3+ cross-validated sources. Low confidence findings are capped at Suggesting.")

    **change-risk-rules.md** — Change assessment rigor and unsafe context handling:
    - Frontmatter: id: change-risk-rules, name: Change Risk Rules, scope: transform_agents, priority: critical
    - Purpose: Explain what goes wrong without change-risk enforcement — changes with unknown blast radius, coupling risk ignored, untested code modified without warning, unsafe changes proceeding to high intervention levels.
    - Rules (covering 2 categories from standards):
      1. Mandatory Blast Radius Assessment — Every remediation must have blast radius assessment with evidence and affected file list. No "this change is isolated" without listing what it touches. (From category 5)
      2. Coupling Analysis Before Structural Changes — Any change that adds, removes, or modifies dependencies/interfaces requires coupling analysis. New dependencies must be justified. (From category 5)
      3. Regression Probability with Evidence — Regression probability must cite test coverage data. Changes to untested code paths require explicit "no test coverage" warning. No "low regression risk" without evidence. (From category 5)
      4. Unsafe Context Downgrade — If any risk dimension score exceeds 4 (out of 5), the change is flagged unsafe and automatically downgraded to Suggesting intervention level. Must explain which dimension and why. (From category 3)
      5. Risk-Recommendation Alignment — overall_risk and recommendation must be logically consistent. High/critical risk cannot produce "proceed" recommendation. Low risk with "reject" recommendation must justify. (New — prevents disconnect between assessment and action)
    - Each rule: statement, rationale, enforcement (concrete mechanism referencing @schema:change-risk and @schema:playbook)
    - DO section: 4+ conforming examples
    - DON'T section: 4+ violations with explanations

    Follow the style and depth of src/rules/epistemic-hygiene.md — substantial Purpose sections, concrete enforcement mechanisms, realistic DO/DON'T examples with explanations. Every rule must be enforceable, not aspirational.
  </action>
