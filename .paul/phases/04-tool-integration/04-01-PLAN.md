<objective>
## Goal
Create 4 tool adapter specifications for the core AEGIS analysis tools: SonarQube, Semgrep, Trivy, and Gitleaks. Each spec defines how to configure, execute, parse output, and normalize signals into the AEGIS signal schema.

## Purpose
Tools are the bridge between raw scanner output and structured agent input. Without tool adapter specs, agents have no standardized way to consume external analysis results. These 4 tools cover the primary signal sources: code quality (SonarQube), static analysis/patterns (Semgrep), vulnerability scanning (Trivy), and secrets detection (Gitleaks). Combined with Plan 04-02's extended tools, this delivers the complete tool integration layer.

## Output
- 4 tool adapter files in `src/tools/`
- Each with 6 sections: Purpose, Configuration, Execution, Output Format, Normalization, Limitations
- Normalization maps to AEGIS signal schema (S-{TTT}-{NNN} format)
- Transform-relevant tools include change-risk signal mapping
</objective>

    - Purpose: Pattern-matching static analysis producing signals for injection flaws, authentication weaknesses, insecure crypto, code quality anti-patterns, framework-specific issues. Broadest domain coverage of any AEGIS tool.
    - Configuration: .semgrep.yml config with rule packs — p/security-audit, p/owasp-top-ten, p/secrets, language-specific packs. Severity filter guidance. Timeout settings for large repos.
    - Execution: Primary: `semgrep scan --config auto --json --output {output_dir}/semgrep-results.json --severity ERROR WARNING --timeout 300 {target_path}`. Docker variant: `docker run --rm -v {target_path}:/src semgrep/semgrep semgrep scan --config auto --json /src`. Parameters table. Runtime: 2-10 min for <100k lines.
    - Output Format: Semgrep JSON results array structure with check_id, path, start/end, extra (message, severity, metadata with cwe/owasp/confidence).
    - Normalization: Map severity (ERROR/WARNING/INFO) → signal severity. Map metadata.cwe → references.cwe. Signal ID pattern: S-SMG-{NNN}. Deduplication note.
    - Limitations: Cannot detect business logic flaws, runtime issues, data flow across ORMs with custom methods. False positives on test fixtures with intentional vulnerabilities, high-entropy non-secret constants.

    **trivy.md** — Trivy (Vulnerability Scanner):
    - Frontmatter: id: trivy, name: Trivy, type: vulnerability_scan, domains_fed: ["04", "05"], install_required: true, install_command: "See Installation section — multiple options: apt, brew, script, Docker"
    - Purpose: Comprehensive vulnerability scanner for OS packages, language dependencies, container images, IaC misconfigurations, and secrets. Primary signal source for dependency CVEs and compliance-relevant vulnerabilities.
    - Configuration: trivy.yaml config for: severity filtering (CRITICAL,HIGH,MEDIUM), ignore file (.trivyignore) for accepted risks, vulnerability database update settings, scan type selection (fs/image/repo)
    - Execution: Primary: `trivy fs --format json --output {output_dir}/trivy-results.json --severity CRITICAL,HIGH,MEDIUM {target_path}`. Docker variant: `docker run --rm -v {target_path}:/target aquasec/trivy fs --format json /target`. Also: `trivy image` variant for container scanning. Installation options: apt-get (Debian/Ubuntu), brew (macOS), install script, Docker image. Parameters table. Runtime: 1-5 min (first run slower due to DB download).
    - Output Format: Trivy JSON structure with Results array containing Target, Vulnerabilities array (VulnerabilityID, PkgName, InstalledVersion, FixedVersion, Severity, Title, Description, References).
    - Normalization: Map severity (CRITICAL/HIGH/MEDIUM/LOW/UNKNOWN) → signal severity. Map VulnerabilityID (CVE-xxx) → references.cve. Signal ID pattern: S-TRV-{NNN}. Note on UNKNOWN severity handling.
    - Limitations: Cannot detect custom/proprietary vulnerabilities, business logic issues, or zero-day exploits not in NVD. False positives on vendored dependencies with backported patches. False negatives on transitive dependencies not in lock files.

    **gitleaks.md** — Gitleaks (Secrets Detection):
    - Frontmatter: id: gitleaks, name: Gitleaks, type: secrets_detection, domains_fed: ["04", "05"], install_required: true, install_command: "See Installation section — multiple options: go install, GitHub releases, brew, Docker"
    - Purpose: Scans git repositories for hardcoded secrets, API keys, tokens, and credentials. Covers both current state and git history. Critical for security (domain 04) and compliance (domain 05).
    - Configuration: .gitleaks.toml config for: custom rules/allowlists, path exclusion patterns (test fixtures, docs with example keys), entropy threshold tuning, custom regex patterns for organization-specific secret formats
    - Execution: Primary: `gitleaks detect --source {target_path} --report-format json --report-path {output_dir}/gitleaks-results.json --verbose`. Docker variant: `docker run --rm -v {target_path}:/target zricethezav/gitleaks detect --source /target --report-format json --report-path /target/.aegis/signals/gitleaks-results.json`. Also: `gitleaks detect --source {target_path} --log-opts="--all"` for full git history scan. Installation: go install (cross-platform), GitHub releases (binary), brew (macOS), apt (Linux), Docker. Parameters table. Runtime: <1 min for current state, 5-30 min for full history.
    - Output Format: Gitleaks JSON array structure with Description, File, StartLine, EndLine, Commit, Secret (redacted), Match, RuleID, Entropy.
    - Normalization: All Gitleaks findings map to signal severity "high" or "critical" (secrets are inherently high-severity). Map RuleID → source_rule. Signal ID pattern: S-GL-{NNN}. Note: blast_radius depends on secret type (API key = service scope, DB credential = data scope).
    - Limitations: Cannot detect secrets in compiled binaries, encrypted config files, or environment variables not committed. False positives on example/documentation keys, high-entropy non-secret strings (UUIDs, hashes), test fixtures. False negatives on custom secret formats not in default rules, secrets committed then removed (only with --log-opts).

    Follow docs/standards/tools.md structure exactly.

## SCOPE LIMITS
- This plan creates 4 core tool adapters only — extended tools (Checkov, Syft, Grype, git-history) are Plan 04-02
- Tool files describe HOW to run tools and normalize output — not WHAT findings mean (that's agent interpretation)
- Normalization maps to signal schema fields — do not invent new schema fields
- Configuration should be practical and concrete but not exhaustive — cover the AEGIS-relevant settings
- SonarQube adapter uses sonar-scanner CLI + REST API as primary interface (universal). MCP server access may be mentioned as an optional enhancement but is NOT the primary integration path.
- All tools must include multiple installation methods (package managers, Docker, binary releases) — AEGIS is a product for diverse environments, not a single-user setup
- All tools should include Docker execution variants alongside native CLI — Docker is the universal fallback
- Do not prescribe severity interpretation — map tool severity to signal severity, agents decide finding severity
