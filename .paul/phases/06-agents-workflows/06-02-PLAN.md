<objective>
## Goal
Create the 7 Core orchestration workflows: one for each AEGIS diagnostic phase (Phases 0-5) plus the disagreement resolution workflow. Each workflow defines step-by-step execution logic for invoking agents, managing session context, validating output, and handling errors — following the workflow convention established in docs/standards/workflows.md.

## Purpose
Workflows are the operational backbone that make agent manifests executable. Without them, agents are static composition files with no invocation logic. Phase orchestration workflows define how each diagnostic phase runs end-to-end — which agents to invoke, what context they receive, how output is validated, and how errors are handled. The disagreement resolution workflow defines how cross-agent conflicts are surfaced, structured, and resolved by the Principal Engineer.

## Output
7 workflow files in `src/core/workflows/`:
- `phase-0-context.md` — Context & Threat Modeling (Principal Engineer solo)
- `phase-1-reconnaissance.md` — Automated Signal Gathering (tool orchestration)
- `phase-2-domain-audits.md` — Deep Domain Audits (8 parallel agent sessions)
- `phase-3-cross-domain.md` — Change Risk, Team Risk & Reality Gap (2 synthesis agents)
- `phase-4-adversarial-review.md` — Devil's Advocate Review (solo adversarial session)
- `phase-5-report.md` — Synthesis & Report Generation (Principal Engineer solo)
- `disagreement-resolution.md` — Structured disagreement resolution protocol
</objective>

    - Single agent: devils-advocate
    - Input: ALL .aegis/findings/*.md from Phases 2-3, ALL .aegis/disagreements/*.md, .aegis/STATE.md with full metrics
    - Devil's Advocate has empty domains[] and tools[] — operates on the complete analytical record
    - Steps: (1) Validate Phase 3 complete, (2) Load complete analytical record, (3) Invoke devils-advocate with all findings + disagreements, (4) Validate critique output against finding and disagreement schemas, (5) Session handoff, (6) Queue disagreement-resolution for any new disagreements raised, (7) Update STATE.md
    - The Devil's Advocate's job is to CHALLENGE — their output creates new disagreements and critiques

    **phase-5-report.md** — Synthesis & Report Generation:
    - Single agent: principal-engineer (Phase 5 role)
    - Input: Complete .aegis/ record — all findings, all disagreements (including Devil's Advocate critique), all resolution records, STATE.md with full phase metrics
    - Steps: (1) Validate Phase 4 complete AND all disagreements have principal_response, (2) Load complete analytical record, (3) Invoke principal-engineer for final synthesis, (4) Validate report sections against report-section schema, (5) Generate final report with all 7 sections (Executive Risk Summary, Architecture Narrative, Findings by Domain, Cross-Validation Notes, Remediation Roadmap, Long-Term Structural Risks, 10x Scale Analysis), (6) Persist report to .aegis/report/, (7) Mark audit as complete in STATE.md
    - CRITICAL: Phase 5 MUST NOT proceed if any disagreement lacks principal_response

    **disagreement-resolution.md** — Disagreement Resolution Protocol:
    - Utility workflow invoked by Phase 2, 3, and 4 workflows when disagreements are detected
    - NOT a phase workflow — it's invoked mid-phase or between phases
    - Input: Disagreement records from .aegis/disagreements/, the findings they reference, agent manifests for involved agents
    - Steps: (1) Load all unresolved disagreements (status: open), (2) For each disagreement: identify the epistemic layer disputed, (3) Classify root cause using the closed taxonomy (threat model mismatch, time horizon mismatch, evidence availability mismatch, risk tolerance mismatch, domain boundary mismatch, optimism vs pessimism bias, tool trust bias), (4) Select applicable resolution model from the 5 canonical models, (5) Present to Principal Engineer for response, (6) Record principal_response (REQUIRED — enforce non-empty), (7) Update disagreement status (mitigated, accepted_risk, deferred, out_of_scope), (8) Write updated records to .aegis/disagreements/
    - Enforce anti-patterns: no auto-resolving, no averaging opinions, no forcing consensus language, no hiding disagreements in footnotes, no treating Devil's Advocate as optional
    - The Principal Engineer's response must: (a) acknowledge explicitly, (b) choose a resolution model, (c) record rationale, (d) assign follow-up if needed

    Avoid:
    - Making disagreement resolution automatic (it requires Principal Engineer judgment)
    - Skipping the principal_response requirement under any circumstance
    - Embedding resolution logic in phase workflows (delegate to disagreement-resolution.md)
