## Goal
Create the 5 Transform agent assembly manifests and 4 Transform workflows — completing the full AEGIS two-system architecture. Transform agents handle Layers B (remediation knowledge) and C (change orchestration). Workflows orchestrate the Transform pipeline (Phases 6-8) plus a safety governance workflow enforcing confidence gating, intervention levels, and the "no auto-execution" boundary.

## Purpose
Without Transform, AEGIS only diagnoses. Transform extends diagnosis into controlled evolution — producing remediation playbooks, risk-scored change plans, guardrails, and PAUL-compatible execution artifacts. The safety workflow is the critical constraint that prevents Transform from overstepping its advisory role.

## Output

- Agents: remediation-architect, change-risk-modeler, pedagogy-agent, guardrail-generator, execution-validator

    Create 4 Transform workflow files following the XML convention. Use Core phase workflows as structural reference.

    **phase-6-remediation.md** — Remediation Synthesis:
    - Sequential agents: remediation-architect THEN pedagogy-agent
    - Input: Complete .aegis/ Layer A record (all findings, disagreements, resolutions, confidence scores)
    - Steps: (1) Validate Core audit complete (Phase 5), (2) Load complete Layer A record, (3) Invoke remediation-architect to group findings by root cause and produce playbooks at all 4 transformation layers, (4) Classify each playbook by intervention level, (5) Invoke transform-safety workflow to validate intervention classifications, (6) Invoke pedagogy-agent to enrich playbooks with educational context, (7) Validate all playbooks against playbook schema, (8) Persist to remediation/ directory, (9) Update STATE.md
    - Output: remediation/playbooks/, remediation/patterns/, remediation/REMEDIATION-SUMMARY.md
    - Error: if finding cannot be remediated (insufficient confidence, unclear root cause) → record as "unremediated" with reason

    **phase-7-risk-validation.md** — Change Risk Validation:
    - Sequential agents: change-risk-modeler THEN guardrail-generator
    - Input: Phase 6 playbooks + codebase structure + .aegis/signals/git-history.md + test coverage signals
    - Steps: (1) Validate Phase 6 complete, (2) Load playbooks + git history + codebase structure, (3) Invoke change-risk-modeler to score every proposed change across 4 dimensions, (4) Invoke transform-safety to check risk thresholds and enforce downgrades, (5) Invoke guardrail-generator to produce project rules from pattern analysis, (6) Establish final risk-adjusted priority ordering, (7) Validate against change-risk schema, (8) Persist to execution/ directory, (9) Update STATE.md
    - Output: execution/risk-scores.yaml, remediation/guardrails/, risk assessment report
    - Error: if risk scoring fails → default to maximum risk, flag for human review

    **phase-8-execution-planning.md** — Execution Planning (PAUL Handoff):
    - Agent: execution-validator
    - Input: Risk-scored change plan from Phase 7 + test infrastructure + deployment config
    - Steps: (1) Validate Phase 7 complete, (2) Load risk-scored plan + codebase context, (3) Invoke execution-validator to define verification steps per change, (4) Build dependency graph for change sequencing, (5) Generate PAUL project artifacts (PROJECT.md, ROADMAP.md, phased plans with risk metadata), (6) Invoke transform-safety for final validation (no auto-execution check), (7) Validate PAUL project structure, (8) Persist to execution/paul-project/, (9) Generate handoff summary, (10) Mark Transform pipeline complete
    - Output: execution/change-graph.yaml, execution/verification-plan.md, execution/paul-project/
    - CRITICAL: AEGIS Transform NEVER applies changes. This workflow produces a plan only.
    - Error: if PAUL generation fails → produce manual remediation plan as fallback

    **transform-safety.md** — Transform Safety Governance:
    - Utility workflow invoked by Phase 6, 7, and 8 workflows at safety checkpoints
    - NOT a phase workflow — it's a validation gate
    - Input: Playbooks/changes being validated + their finding confidence + risk scores
    - Steps:
      (1) **Confidence gating:** For each playbook/change, verify finding confidence meets minimum for declared intervention level (Suggesting=Low/1 source, Planning=Medium/2 sources, Authorizing=High/3+ sources, Executing=High/3+ with cross-validation)
      (2) **Risk threshold check:** For each change with risk scores, check all 4 dimensions. If ANY dimension exceeds "high" threshold → force downgrade to Suggesting, explain why
      (3) **Intervention level enforcement:** Verify no playbook claims a higher intervention level than its confidence warrants. Immutable after approval.
      (4) **No auto-execution boundary:** Verify Transform output is plans only — no execution commands, no direct file modifications, no bypass mechanisms
      (5) **Refusal conditions check:** Verify none of the 5 refusal conditions are met: (a) confidence insufficient, (b) risk exceeds bounds with no downgrade possible, (c) insufficient test coverage, (d) unresolved high-severity disagreements, (e) changes outside audit scope. If any condition met → refuse to generate remediation for affected findings.
      (6) **Record safety audit:** Log all safety checks performed, any downgrades applied, any refusals issued
    - Error: Safety violations are NOT errors to handle — they are working correctly. A refusal IS the correct behavior.

    Avoid:
    - Any workflow step that executes changes (Transform produces plans only)
    - Skipping transform-safety invocation (it's mandatory at every phase boundary)
    - Embedding domain knowledge in Transform workflows (delegate to agents)
  </action>
