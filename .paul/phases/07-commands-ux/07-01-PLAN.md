<objective>
## Goal
Create the 4 Core slash commands that serve as user entry points for AEGIS diagnostic audits: `/aegis:audit`, `/aegis:resume`, `/aegis:status`, and `/aegis:report`. Each command is a guided wizard experience that delegates to Core workflows.

## Purpose
Commands are the ONLY part of AEGIS users interact with directly. Without them, all the agents, workflows, domains, schemas, and tools are inaccessible. This plan makes the Core diagnostic pipeline usable by providing the guided UX layer.

## Output
4 command files in `src/core/commands/` following the command convention (YAML frontmatter + 5 XML sections).
</objective>

  <action>
    Create `src/core/commands/` directory and all 4 command files.

    **All commands share this structure:**
    - YAML frontmatter: `name`, `description`, `argument-hint` (optional)
    - 5 XML sections: `<objective>`, `<execution_context>`, `<context>`, `<process>`, `<success_criteria>`

    **audit.md** (`/aegis:audit`):
    - argument-hint: `"[path-to-repo]"`
    - The master command — primary entry point for starting audits
    - Process flow:
      1. Determine target (argument or cwd)
      2. Check for existing .aegis/ — if found, route to /aegis:resume
      3. Initialize new audit: detect languages/frameworks, display repo info
      4. Scope selection: Full (all phases, all domains) / Targeted (select domains) / Quick scan (Phases 0-2 only)
      5. Audit configuration: which tools to enable/disable, agent scope overrides
      6. Confirmation: display audit plan (scope, phases, domains, estimated sessions), confirm/modify/cancel
      7. Initialize .aegis/ directory (STATE.md + MANIFEST.md)
      8. Delegate to phase-0-context workflow
    - execution_context references: all 6 phase workflows + session-handoff + disagreement-resolution
    - context references: $ARGUMENTS, .aegis/STATE.md, .aegis/MANIFEST.md
    - Every decision point has a Cancel option

    **resume.md** (`/aegis:resume`):
    - argument-hint: `"[phase-number]"`
    - Resumes an interrupted audit from last checkpoint
    - Process flow:
      1. Check .aegis/STATE.md exists — if not, route to /aegis:audit
      2. Read STATE.md: extract current phase, finding counts, last action, next action
      3. Display progress: phase-by-phase table with status indicators
      4. Present options: Resume from checkpoint (recommended) / Re-run specific phase / Start fresh (warning: deletes state) / Cancel
      5. Delegate to appropriate phase workflow based on selection
    - execution_context references: all 6 phase workflows
    - context references: $ARGUMENTS, .aegis/STATE.md

    **status.md** (`/aegis:status`):
    - No argument-hint (reads current state)
    - Read-only — displays audit state without modifying anything
    - Process flow:
      1. Check .aegis/STATE.md exists — if not, report "No active audit"
      2. Display: audit target, start time, current phase, overall status
      3. Phase progress table: phase number, name, status, agent(s), finding count, timestamps
      4. Summary stats: total findings, disagreements (open/resolved), domains covered
      5. Next action suggestion: route to /aegis:resume or /aegis:report as appropriate
    - execution_context: none (read-only, no workflow delegation)
    - context references: .aegis/STATE.md

    **report.md** (`/aegis:report`):
    - argument-hint: `"[format]"`
    - Generates or regenerates the final diagnostic report
    - Process flow:
      1. Check .aegis/STATE.md — verify Phase 5 complete (or all Core phases complete)
      2. If Phase 5 incomplete, show warning with phase progress and route to /aegis:resume
      3. Check existing report: if .aegis/report/ exists, offer regenerate or view options
      4. Report options: Full report (all sections) / Executive summary only / Findings by domain only / Disagreements only
      5. Delegate to phase-5-report workflow for generation
      6. Display report location and next steps (archive, Transform pipeline)
    - execution_context references: phase-5-report workflow
    - context references: $ARGUMENTS, .aegis/STATE.md, .aegis/report/

    **Avoid and WHY:**
    - Do NOT embed tool execution logic — commands are wizard UX, not execution engines (anti-pattern from convention)
    - Do NOT skip cancellation options at decision points — users must never feel trapped
    - Do NOT hardcode tool names or non-standard paths — reference workflows and .aegis/ conventions only
    - Do NOT duplicate workflow step details — commands route to workflows, they don't replicate internal logic
