<objective>
## Goal
Validate the complete AEGIS specification set (90 files across 8 component types) for internal consistency, cross-reference integrity, convention compliance, and compositional correctness. Produce a version-locked manifest and validation report that confirms v0.1 readiness.

## Purpose
AEGIS is a specification-phase project — 90 markdown files describing a multi-agent audit system. Before declaring v0.1 complete, every cross-reference must resolve, every file must follow its component convention, and every agent must correctly compose its persona + domains + tools + schemas + rules. This validation replaces a "live run" (which requires runtime implementation beyond v0.1 scope) with a rigorous structural verification.

## Output
- Cross-reference integrity report (all inter-component references validated)
- Convention compliance report (all 90 files checked against Phase 1 standards)
- Version-lock manifest (SHA-256 hashes for all spec files)
- Validation summary (pass/fail per component type, issues found, v0.1 readiness)
- README.md updated with final v0.1 metrics
</objective>

  <action>
    Validate all 90 files against their component type conventions from docs/standards/:

    1. **Domains (14 files):** YAML frontmatter has id, number, name, owner_agents. Body has all 8 required sections in order (Overview, Audit Questions, Failure Patterns, Best Practice Patterns, Red Flags, Tool Affinities, Standards & Frameworks, Metrics). Verify 1:1 failure/best-practice mapping.

    4. **Tools (8 files):** Frontmatter has id, name, type, domains_fed, install_required. Body has Purpose, Configuration, Execution, Output Format, Normalization, Limitations.
    5. **Personas (17 files — 12 core + 5 transform):** Frontmatter has id, name, role, active_phases. Body has 8 XML sections: identity, mental_models, risk_philosophy, thinking_style, triggers, argumentation, confidence_calibration, constraints.
    6. **Agents (17 files — 12 core + 5 transform):** Frontmatter has id, name, persona, domains, tools, schemas, rules, active_phases, parallel_eligible. Body has Assembly Notes, Session Context.
    7. **Workflows (11 files — 8 core + 3 transform):** XML structure with purpose, phase_context, required_input, process (with named steps), output, error_handling.
    8. **Commands (8 files — 4 core + 4 transform):** Frontmatter has name, description, argument-hint. Body has 5 XML sections: objective, execution_context, context, process, success_criteria.

    Use parallel subagents (one per component type) to maximize efficiency.
    Report: file path, pass/fail, list of deviations with severity (blocker/warning/cosmetic).
