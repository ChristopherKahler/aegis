<objective>
## Goal
Create an interactive install script that copies the AEGIS framework to `~/.claude/aegis/` and commands to `~/.claude/commands/aegis/`, then walks the user through installing each OSS analysis tool with Y/N prompts.

## Purpose
Phase 9 made all files reference `@~/.claude/aegis/` paths, but there's no mechanism to actually place the files there. Without an install script, users must manually copy 82 framework files and 8 commands to the right locations. This phase bridges from "installable file structure" to "actually installed."

## Output
- `install.sh` — Single interactive install script at repo root
</objective>

## AC-1: Framework Files Placed
```gherkin
Given the AEGIS repo is cloned locally
When the user runs `bash install.sh`
Then all files under `src/` are copied to `~/.claude/aegis/` preserving directory structure
And `~/.claude/aegis/` contains the same directory tree as `src/` (core/, transform/, domains/, schemas/, rules/, tools/)
```

## AC-2: Commands Placed
```gherkin
Given the install script runs
When the framework copy completes
Then all 8 `.md` files from `commands/` are copied to `~/.claude/commands/aegis/`
And each command is accessible as `/aegis:{name}` in Claude Code
```

## AC-3: Interactive Tool Installation
```gherkin
Given the framework and commands are placed
When the install script reaches the tool installation phase
Then it prompts Y/N for each OSS tool: SonarQube, Semgrep, Trivy, Gitleaks, Checkov, Syft, Grype
And for each Y: attempts to install via the best available method (checking brew, pip, apt, go, docker)
And for each N: skips with a note that the tool can be installed later
And git-history (built-in git commands) is noted as always available — no install needed
```

## AC-4: Post-Install Verification
```gherkin
Given tool installation completes (Y or N for each)
When the install script reaches the verification phase
Then it tests each installed tool by running its version/help command
And reports pass/fail for each tool
And provides a summary: X/7 tools installed, Y/7 tools skipped
```

## AC-5: Idempotent and Safe
```gherkin
Given AEGIS is already installed at ~/.claude/aegis/
When the user runs install.sh again
Then it warns that existing installation found
And asks to overwrite or cancel
And does not corrupt partial state on failure
```

<tasks>

<task type="auto">
  <name>Task 1: Create install.sh</name>
  <files>install.sh</files>
  <action>
    Create a bash script at repo root with these sections:

    **1. Header and environment:**
    - Shebang: `#!/usr/bin/env bash`
    - `set -euo pipefail` for safety
    - Define constants: `AEGIS_HOME=~/.claude/aegis`, `AEGIS_COMMANDS=~/.claude/commands/aegis`, `SCRIPT_DIR` (directory of install.sh)
    - Color helpers for output formatting (green/yellow/red/bold)

    **2. Pre-flight checks:**
    - Verify `~/.claude/` exists (Claude Code installed)
    - Verify `src/` and `commands/` exist relative to script (running from repo root)
    - Check for existing installation — if `$AEGIS_HOME` exists, prompt: "Existing AEGIS installation found. Overwrite? [y/N]"

    **3. Framework installation:**
    - Create `$AEGIS_HOME` directory
    - Copy `src/` contents to `$AEGIS_HOME/` preserving structure: `cp -r src/* $AEGIS_HOME/`
    - Report count: "Copied N framework files to ~/.claude/aegis/"

    **4. Commands installation:**
    - Create `$AEGIS_COMMANDS` directory
    - Copy `commands/*.md` to `$AEGIS_COMMANDS/`
    - Report: "Installed 8 slash commands to ~/.claude/commands/aegis/"

    **5. Tool installation (interactive):**
    - Header: "AEGIS uses 7 OSS analysis tools. Install now? You can always install later."
    - Note: "git-history uses built-in git commands — always available."
    - For each tool, prompt with: `[tool-name] — [brief description]. Install? [y/N]`
    - Tools and their install methods (try in order, use first available):
      - **SonarQube** (special — requires server + scanner, most complex tool):
        - First, display disclaimer block:
          ```
          ┌─────────────────────────────────────────────────────┐
          │  SONARQUBE — Important Context                      │
          │                                                     │
          │  Unlike other AEGIS tools, SonarQube requires TWO   │
          │  components:                                        │
          │                                                     │
          │  1. Scanner CLI — analyzes your code                │
          │  2. Server — stores rules, runs analysis, serves    │
          │     results via API                                 │
          │                                                     │
          │  The scanner CANNOT work without a server.          │
          │  Server options: Docker (local) or SonarQube Cloud. │
          │                                                     │
          │  SonarQube provides:                                │
          │  • Code complexity metrics (cyclomatic, cognitive)  │
          │  • Duplication detection                            │
          │  • Code smell classification                        │
          │  • Quality gate enforcement                         │
          │  • Technical debt estimation                        │
          │                                                     │
          │  These are NOT covered by any other AEGIS tool.     │
          │  Semgrep covers security/correctness overlap, but   │
          │  NOT complexity, duplication, or debt metrics.       │
          └─────────────────────────────────────────────────────┘
          ```
        - Prompt: "Install SonarQube? [y/N]"
        - **If N (skip):** Show confirmation:
          ```
          ⚠ Skipping SonarQube means AEGIS loses:
            • Code complexity analysis (cyclomatic, cognitive)
            • Duplication detection across codebase
            • Code smell classification and debt estimation
            • Quality gate enforcement

          Domains 01 (Architecture), 03 (Correctness), 06 (Testing),
          09 (Maintainability) will have REDUCED signal coverage.
          Semgrep still covers security and pattern-based analysis.

          You can install SonarQube later by re-running install.sh.
          ```
          Then: "Are you sure you want to skip SonarQube? [y/N]"
          - If confirmed skip: move on
          - If N (changed mind): fall through to install flow
        - **If Y (install):** Sub-prompt:
          ```
          SonarQube server options:

          [1] Docker (local)
              • Runs sonarqube:community on localhost:9000
              • Requires Docker installed and running
              • ~800MB image download
              • Server must be running during audits

          [2] SonarQube Cloud
              • Uses cloud.sonarqube.com (free for public repos)
              • Requires account + authentication token
              • No local server needed
              • Token configuration guided after install
          ```
          Then: "Choose [1/2]:"
          - **If Docker chosen:**
            - Check `command -v docker` — if not found, warn "Docker not found. Install Docker first, then re-run install.sh." and skip SonarQube
            - Check Docker daemon running (`docker info`) — if not, warn and skip
            - `docker pull sonarqube:community`
            - Install scanner CLI: `npm install -g sonar-scanner` || `brew install sonar-scanner`
            - Note: "Start server with: docker run -d --name sonarqube -p 9000:9000 sonarqube:community"
            - Note: "Server must be running before AEGIS audit. Default credentials: admin/admin"
          - **If Cloud chosen:**
            - Install scanner CLI only: `npm install -g sonar-scanner` || `brew install sonar-scanner`
            - Guide: "To configure SonarQube Cloud:"
            - "1. Create account at https://sonarcloud.io"
            - "2. Generate token at https://sonarcloud.io/account/security"
            - "3. Set environment variable: export SONAR_TOKEN=your-token"
            - "4. Create sonar-project.properties in your target project"
            - Note: "Run /aegis:validate after setup to verify connection"
      - **Semgrep**: `pip install semgrep` || `brew install semgrep` || Docker fallback
      - **Trivy**: `brew install trivy` || `curl -sfL https://raw.githubusercontent.com/aquasecurity/trivy/main/contrib/install.sh | sh -s -- -b /usr/local/bin` || Docker fallback
      - **Gitleaks**: `brew install gitleaks` || `go install github.com/gitleaks/gitleaks/v8@latest` || Docker fallback
      - **Checkov**: `pip install checkov` || `brew install checkov` || Docker fallback
      - **Syft**: `curl -sSfL https://raw.githubusercontent.com/anchore/syft/main/install.sh | sh -s -- -b /usr/local/bin` || `brew install syft`
      - **Grype**: `curl -sSfL https://raw.githubusercontent.com/anchore/grype/main/install.sh | sh -s -- -b /usr/local/bin` || `brew install grype`
    - Track installed/skipped lists

    **6. Post-install verification:**
    - For each tool that was installed (Y), test accessibility:
      - `sonarqube`: `sonar-scanner --version` (scanner CLI) + `docker image inspect sonarqube:community` (if Docker server chosen)
      - `semgrep`: `semgrep --version`
      - `trivy`: `trivy --version`
      - `gitleaks`: `gitleaks version`
      - `checkov`: `checkov --version`
      - `syft`: `syft version`
      - `grype`: `grype version`
    - Report pass/fail per tool
    - Always verify: `git --version` (git-history)

    **7. Summary:**
    ```
    ════════════════════════════════════════
    AEGIS Installation Complete
    ════════════════════════════════════════

    Framework: ~/.claude/aegis/ (N files)
    Commands:  ~/.claude/commands/aegis/ (8 commands)

    Tools:
      ✓ semgrep (installed)
      ✓ trivy (installed)
      ✗ sonarqube (skipped)
      ...

    Next: Run /aegis:init in your target project.
    ════════════════════════════════════════
    ```

    **Key design decisions:**
    - Default to N (no install) for each tool — conservative, no surprises
    - Use `command -v` to detect available package managers (brew, pip, go, docker)
    - Do NOT use sudo — tools install to user-accessible locations
    - Each tool install is independent — one failure doesn't block others
    - The script does NOT install Docker itself — just uses it if available
    - Avoid: complex dependency resolution, auto-detecting OS distro, requiring root
