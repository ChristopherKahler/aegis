  <name>Task 1: Create docs/GETTING-STARTED.md</name>
  <files>docs/GETTING-STARTED.md</files>
  <action>
    Create `docs/GETTING-STARTED.md` with the following structure:

    **Title:** "Getting Started with AEGIS"
    **Subtitle:** "From zero to first audit in 10 minutes"

    **Sections (in order):**

    1. **Prerequisites**
       - Required: Claude Code CLI installed (`claude --version` to verify), git
       - Optional: Docker (for SonarQube local server)
       - Note: AEGIS installs its own analysis tools — you don't need Semgrep, Trivy, etc. pre-installed

    2. **Install AEGIS** (two methods)
       - One-command: `curl -sSL https://raw.githubusercontent.com/ChristopherKahler/aegis/main/install.sh | bash`
       - Clone and install: `git clone ... && cd aegis && bash install.sh`
       - Show example output snippet: framework files copied, tools installed, summary
       - Note: installer is interactive — walks through each tool with Y/N

    3. **Verify Installation**
       - Run `/aegis:validate` in Claude Code
       - Show example output: tool pass/fail table
       - What to do if tools show "not found" (re-run installer or check PATH)

    4. **Initialize a Project**
       - Navigate to the repo you want to audit
       - Run `/aegis:init`
       - Show example output: .aegis/ created with STATE.md, MANIFEST.md
       - Note: this creates audit workspace, doesn't start the audit

    5. **Run Your First Audit**
       - Run `/aegis:audit`
       - Walk through the wizard: scope selection (recommend Full for first run), tool selection
       - Show example confirmation output
       - Explain: Phase 0 starts automatically after confirmation

    6. **Multi-Session Workflow**
       - Explain: audits span multiple sessions (6+ phases)
       - After each phase: checkpoint with continue/pause/abort
       - If you pause: run `/aegis:resume` to pick up where you left off
       - Run `/aegis:status` anytime to see progress
       - Show example checkpoint output

    7. **Viewing Results**
       - After Phase 5 completes: run `/aegis:report` to generate the audit report
       - Report location: `.aegis/report/AEGIS-REPORT.md`
       - Optional next step: `/aegis:transform` for remediation pipeline

    8. **Command Reference**
       - Quick table of all 10 commands with one-line descriptions:
         - `/aegis:audit` — Start a diagnostic audit
         - `/aegis:resume` — Resume a paused audit
         - `/aegis:status` — View audit progress (read-only)
         - `/aegis:report` — Generate the final audit report
         - `/aegis:init` — Initialize AEGIS in a project
         - `/aegis:validate` — Test tool installation
         - `/aegis:transform` — Start remediation pipeline
         - `/aegis:remediate` — Generate remediation plans
         - `/aegis:playbook` — Create a remediation playbook
         - `/aegis:guardrails` — Generate project rules from findings

    9. **Troubleshooting**
       - "Tool X not found" → Re-run `bash install.sh`, check `~/.local/bin` in PATH, check venv at `~/.local/share/aegis/venvs/`
       - "Permission denied during install" → AEGIS never needs sudo; check file permissions in `~/.local/bin/`
       - "Docker not running" (SonarQube) → Start Docker, or skip SonarQube (other tools work without it)
       - "Context window exceeded" → Audit phases are designed for 200k context; if hitting limits, use targeted audit (fewer domains)
       - ".aegis/ already exists" → `/aegis:init` offers resume, fresh start, or cancel
       - "How do I update AEGIS?" → Re-run the install script — it overwrites framework files and preserves tool installations

    **Style guidelines:**
    - Use code blocks for all commands and example output
    - Keep example outputs realistic but concise (5-10 lines max per example)
    - No emojis
    - Direct, practical tone — this is a reference doc, not a tutorial
    - Link back to README.md for architecture details

    **Do NOT:**
    - Duplicate the full README content (link to it instead)
    - Include internal implementation details (no mention of schemas, rules, personas by name)
    - Make promises about output quality or accuracy
